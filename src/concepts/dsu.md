# 유니온 파인드 (Union-Find) / 서로소 집합 (DSU)

## 한 줄 요약
DSU(Disjoint Set Union, 서로소 집합)는 "여러 원소를 그룹(집합)으로 묶고", "두 원소가 같은 그룹인지 빠르게 확인"하는 데 특화된 자료구조다. 핵심 연산은 단 두 개: `union(a, b)`(두 그룹 합치기), `find(x)`(x가 속한 그룹의 대표 찾기).

---

## 언제 쓰나
실제 코딩테스트에서 아래 신호가 보이면 DSU를 떠올리면 된다.

- **"A와 B가 연결되어 있는가?"** 를 여러 번 물어보는 문제 (네트워크 연결, 친구 관계, 같은 팀인지 판별)
- **그래프의 연결 요소(Connected Component) 개수**를 세는 문제
- **사이클 판별** — 간선을 추가할 때 두 끝점이 이미 같은 집합이면 사이클이 생긴다
- **최소 스패닝 트리(MST)의 크루스칼(Kruskal) 알고리즘** — 간선을 가중치 순으로 고르며 사이클을 피할 때 DSU로 판정
- **동적으로 그룹이 합쳐지는 상황** — 섬 합치기, 계정 병합, 집합 분류
- 좌표/노드를 **그룹핑**해서 "몇 덩어리인가", "이 둘이 같은 덩어리인가"를 묻는 문제 전반

대표 백준 문제: 1717(집합의 표현), 1976(여행 가자), 4195(친구 네트워크), 1197(최소 스패닝 트리, 크루스칼).

---

## 왜 이걸 쓰나 + 대안과 차이

"두 노드가 연결되었는지" 판별만 보면 BFS/DFS로도 가능하다. 하지만 차이가 크다.

| 방법 | 연결 여부 1회 질의 | 그룹이 계속 합쳐질 때 | 특징 |
|------|----------------|------------------|------|
| **BFS/DFS** | O(V+E) 매번 탐색 | 합칠 때마다 다시 탐색해야 함 → 비효율 | 경로 자체가 필요할 때 유리 |
| **DSU** | 거의 O(1) (α) | union으로 즉시 합침, 누적 효율적 | 경로는 모름, "같은 집합인지"만 빠름 |

- **DSU는 "합치기"와 "같은 집합인지 묻기"가 섞여서 수없이 반복될 때** 압도적으로 빠르다. 간선이 동적으로 늘어나도 매번 전체 탐색을 하지 않는다.
- **단, DSU는 한 번 합친 집합을 다시 쪼갤 수 없다**(분리 연산이 기본 제공되지 않음). "끊기"가 필요하면 DSU가 부적합하거나, 오프라인으로 역방향 처리하는 트릭이 필요하다.
- **실제 경로(어떻게 연결됐는지)** 를 알아야 하면 BFS/DFS가 필요하다. DSU는 "대표(루트)"만 알려줄 뿐 경로 정보를 주지 않는다.

핵심 아이디어는 각 집합을 **트리**로 표현하고, 트리의 **루트(root)** 를 그 집합의 "대표 ID"로 삼는 것이다. 두 원소의 루트가 같으면 같은 집합이다.

---

## 동작 과정

`parent[]` 배열만으로 트리를 표현한다. `parent[i]`는 i의 부모 노드이며, `parent[i] == i`이면 i가 루트(대표)다.

### 1) 초기 상태 — 모든 원소가 자기 자신이 대표 (각자 독립 집합)

```
원소 :  0   1   2   3   4   5
parent: [0] [1] [2] [3] [4] [5]   <- 자기 자신을 가리킴 = 각자 루트

트리 모양:
  0    1    2    3    4    5     (전부 따로 떨어진 6개 집합)
```

### 2) union(0, 1) — 0과 1을 합친다

1의 루트(1)를 0의 루트(0) 밑으로 붙인다.

```
parent: [0] [0] [2] [3] [4] [5]
                 ^
                1의 부모가 0이 됨

트리 모양:
   0        2    3    4    5
   |
   1
```

### 3) union(2, 3), union(0, 2) — 차례로 합치기

```
union(2,3):
parent: [0] [0] [2] [2] [4] [5]
   0      2       4    5
   |      |
   1      3

union(0,2):  (0의 루트=0, 2의 루트=2 → 2를 0 밑으로)
parent: [0] [0] [0] [2] [4] [5]
        0
       /|
      1 2
        |
        3
```

### 4) find(3) — 3의 대표는 누구? + 경로 압축(Path Compression)

```
find(3) 추적:  3 -> parent[3]=2 -> parent[2]=0 -> parent[0]=0 (루트!)
=> 3이 속한 집합의 대표는 0

[경로 압축 적용 후]  탐색하며 지나간 노드들을 루트에 직접 연결!
parent: [0] [0] [0] [0] [4] [5]
                     ^
              3의 부모를 곧장 0으로 변경

트리가 납작해짐 (다음 find가 훨씬 빨라짐):
        0
      / | \
     1  2  3
```

### 5) 같은 집합인지 판정

```
isSame(1, 3)?  find(1)=0,  find(3)=0  ->  0 == 0  ->  YES (같은 집합)
isSame(1, 5)?  find(1)=0,  find(5)=5  ->  0 != 5  ->  NO  (다른 집합)
```

이렇게 **find로 루트를 찾고, union으로 루트끼리 붙이며, 경로 압축으로 트리를 납작하게** 유지하는 것이 DSU의 전부다.

## C++ 예제 코드

두 가지 최적화를 모두 적용한 표준 구현이다.
- **경로 압축(Path Compression)**: find 도중 지나간 노드를 루트에 바로 붙임
- **랭크/사이즈 기반 합치기(Union by Size)**: 작은 트리를 큰 트리 밑에 붙여 높이를 낮게 유지

```cpp
#include <bits/stdc++.h>
using namespace std;

struct DSU {
    vector<int> parent; // parent[i] = i의 부모 (parent[i]==i 이면 루트)
    vector<int> sz;     // sz[루트] = 그 집합의 크기

    DSU(int n) {
        parent.resize(n);
        sz.assign(n, 1);                 // 처음엔 각 집합 크기 1
        for (int i = 0; i < n; i++)
            parent[i] = i;               // 각자 자신이 대표
    }

    // x가 속한 집합의 루트(대표)를 찾는다 + 경로 압축
    int find(int x) {
        if (parent[x] == x) return x;          // 자기가 루트면 그대로 반환
        return parent[x] = find(parent[x]);    // 결과를 부모로 직접 저장(압축)
    }

    // a와 b가 속한 두 집합을 합친다. 새로 합쳐졌으면 true, 이미 같으면 false
    bool unite(int a, int b) {
        int ra = find(a), rb = find(b);
        if (ra == rb) return false;            // 이미 같은 집합 -> 합칠 필요 없음
        if (sz[ra] < sz[rb]) swap(ra, rb);     // 항상 큰 집합(ra) 밑에 붙이기
        parent[rb] = ra;                       // 작은 트리 루트를 큰 트리 루트에 연결
        sz[ra] += sz[rb];                      // 크기 갱신
        return true;
    }

    // a와 b가 같은 집합에 속하는가?
    bool same(int a, int b) {
        return find(a) == find(b);
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    // 예시: 원소 6개(0~5), 질의를 처리
    DSU dsu(6);

    dsu.unite(0, 1);
    dsu.unite(2, 3);
    dsu.unite(0, 2);

    cout << (dsu.same(1, 3) ? "YES" : "NO") << "\n"; // YES (0-1-2-3 한 덩어리)
    cout << (dsu.same(1, 5) ? "YES" : "NO") << "\n"; // NO  (5는 따로)

    // 연결 요소(집합) 개수 세기: 루트인 원소의 수
    int components = 0;
    for (int i = 0; i < 6; i++)
        if (dsu.find(i) == i) components++;
    cout << "components = " << components << "\n"; // {0,1,2,3}, {4}, {5} -> 3

    return 0;
}
```

크루스칼 MST에서의 활용 예 (개념):

```cpp
// 간선들을 가중치 오름차순 정렬 후
sort(edges.begin(), edges.end()); // {weight, u, v}
DSU dsu(n);
long long total = 0;
for (auto [w, u, v] : edges) {
    if (dsu.unite(u, v))  // 사이클을 만들지 않을 때만 채택
        total += w;       // 다른 집합이었다면 간선 선택
}
// total = 최소 스패닝 트리의 가중치 합
```

## 시간복잡도

- **경로 압축 + union by size/rank 둘 다 적용 시**: 한 연산당 평균 **O(α(N))** (애커만 함수의 역함수).
  - α(N)은 N이 우주의 원자 수만큼 커도 4 이하다. 실질적으로 **상수 시간 O(1)** 으로 봐도 무방하다.
- **둘 중 하나만 적용**: O(log N) 수준.
- **아무 최적화도 안 하면**: 트리가 한 줄(연결 리스트)로 늘어져 최악 **O(N)** 까지 나빠진다 → 반드시 최적화하자.
- 공간복잡도: parent, size 배열로 **O(N)**.

## 흔한 실수 / 팁

- **`parent[x]`와 `find(x)`를 혼동하지 말 것.** `parent[x]`는 바로 위 부모일 뿐, 대표(루트)가 아니다. 같은 집합인지 비교할 때는 반드시 `find()`로 루트끼리 비교한다.
- **union할 때 루트끼리 붙여라.** `parent[b] = a`처럼 원소를 그대로 붙이면 안 되고, `parent[find(b)] = find(a)` 형태여야 한다. 위 예제는 미리 루트를 구해서 처리한다.
- **경로 압축을 빼먹으면** 깊은 트리에서 find가 느려져 시간 초과가 난다. 재귀 `return parent[x] = find(parent[x]);` 한 줄을 꼭 넣자.
- **재귀 깊이 주의.** N이 매우 크고 최적화가 부족하면 재귀 find가 스택 오버플로를 낼 수 있다. 불안하면 반복문(while) 버전으로 작성하거나 union by size를 반드시 함께 쓴다.
- **인덱스 범위/오프셋.** 노드가 1번부터 시작하면 `DSU(n+1)`로 잡거나, 이름(문자열)이면 map으로 정수 ID를 부여한 뒤 사용한다(예: 친구 네트워크 문제).
- **집합 크기·개수가 필요하면 size 배열을 같이 관리.** 합칠 때 `sz[ra] += sz[rb]`로 갱신하면 "현재 그룹 인원수" 같은 질의를 O(α)에 답할 수 있다.
- **분리(쪼개기)는 기본 DSU로 불가능.** "연결을 끊는" 쿼리가 있으면 쿼리를 거꾸로(오프라인) 처리하거나 다른 자료구조를 고려한다.
- **같은 집합인데 또 union 호출**해도 안전하다(`unite`가 false 반환). 사이클 판정/MST에서 이 반환값을 그대로 활용하면 깔끔하다.
