# 구현 (Implementation / Simulation)

## 한 줄 요약
"구현"은 특별한 알고리즘 트릭 없이, **문제에서 시킨 그대로의 절차를 코드로 정확하게 옮기는** 능력을 묻는 유형이다. 즉 "머릿속으로 푸는 것"과 "코드로 한 줄 한 줄 빠짐없이 적는 것" 사이의 거리를 좁히는 일이다.

---

## 언제 쓰나
구현은 별도의 자료구조 이름이 붙은 유형이라기보다, "문제 설명을 그대로 따라 하면 답이 나오는" 거의 모든 문제를 포함한다. 대표적인 문제 유형은 다음과 같다.

- **시뮬레이션**: 로봇/캐릭터가 명령(상/하/좌/우)에 따라 격자판 위를 이동하는 문제
- **완전 탐색(브루트 포스)**: 가능한 모든 경우를 직접 다 만들어 보고 조건에 맞는 것을 세는 문제 (예: 모든 시각 중 '3'이 포함된 횟수 세기)
- **격자(2D 배열) 처리**: 지도에서 특정 규칙으로 색을 칠하거나, 회전시키거나, 인접 칸을 확인하는 문제
- **문자열 파싱/포맷팅**: 입력 형식을 분해하고 규칙대로 다시 조립하는 문제 (예: 날짜 계산, 좌표 변환 "a1" → (0,0))
- **달력/시간 계산**: 윤년, 요일, 분/초 단위 누적 등
- **게임 규칙 재현**: 뱀, 테트리스, 청소 로봇처럼 정해진 규칙대로 상태를 갱신하는 문제

> 한 줄 정리: "이거 그냥 시키는 대로 하면 되는 거 아냐?" 싶으면 그게 바로 구현 문제다.

---

## 왜 이걸 쓰나 + 대안과 차이
구현 유형은 "더 빠른 알고리즘을 찾아야 하나?"를 고민하기 전에 풀 수 있는 경우가 많다. 입력 크기가 작아서 굳이 최적화가 필요 없기 때문이다.

| 접근 | 핵심 질문 | 언제 |
|------|-----------|------|
| **구현 / 완전 탐색** | "그냥 다 해보면 되지 않나?" | N이 작다 (예: N ≤ 100, 경우의 수 ~10^7 이하) |
| **그리디** | "매 순간 최선을 고르면 전체 최선?" | 그렇게 골라도 정답이 보장되는 구조일 때 |
| **DP** | "겹치는 부분 문제를 저장해 재활용?" | 경우의 수가 폭발하지만 상태가 겹칠 때 |
| **그래프 탐색(BFS/DFS)** | "연결 관계를 따라가야 하나?" | 칸/노드 사이 이동·연결이 핵심일 때 |

**핵심 차이**: 그리디·DP는 "어떻게 하면 더 똑똑하게 줄일까"가 본질이라면, 구현은 **"실수 없이 끝까지 정확하게 적을 수 있는가"** 가 본질이다. 알고리즘 난이도는 낮지만, 조건이 많아 코드가 길어지고 사소한 실수(인덱스, 경계, 부등호)로 틀리기 쉽다.

또한 완전 탐색은 구현의 한 종류인데, "경우의 수가 감당 가능한 크기인지"를 먼저 계산하는 습관이 중요하다. 예를 들어 24시간 × 60분 × 60초 = 86,400가지는 전부 다 돌려도 순식간이므로 완전 탐색이 정답이다.

## 동작 과정
예시 문제: **상하좌우(시뮬레이션)**
- N×N 격자(좌상단 (1,1) ~ 우하단 (N,N))에서 시작점은 (1,1).
- 명령 문자열을 받아 한 글자씩 처리한다: `R`(오른쪽), `L`(왼쪽), `U`(위), `D`(아래).
- **격자 밖으로 나가는 이동은 무시**한다.
- 모든 명령을 처리한 뒤 최종 좌표를 출력한다.

여기서는 N=5, 명령 = `R R R U D D` 라고 하자. (행=row, 열=col, 1부터 시작)

```
초기 상태: 위치 (r=1, c=1)
좌표계:  c→ 1 2 3 4 5
       r 1 [S]. . . .
       ↓ 2  . . . . .
         3  . . . . .
         4  . . . . .
         5  . . . . .

[1] 명령 'R' (오른쪽: c+1)  ->  (1,1)→(1,2)  OK
         1  . [X] . . .
         2  . . . . .
         (격자 안 → 이동 확정, 위치=(1,2))

[2] 명령 'R'  ->  (1,2)→(1,3)  OK
         1  . . [X] . .
         (위치=(1,3))

[3] 명령 'R'  ->  (1,3)→(1,4)  OK
         1  . . . [X] .
         (위치=(1,4))

[4] 명령 'U' (위: r-1)  ->  (1,4)→(0,4)  ✗ r<1 이므로 격자 밖!
         => 이동 무시, 위치 그대로 (1,4)
         1  . . . [X] .

[5] 명령 'D' (아래: r+1)  ->  (1,4)→(2,4)  OK
         1  . . . . .
         2  . . . [X] .
         (위치=(2,4))

[6] 명령 'D'  ->  (2,4)→(3,4)  OK
         2  . . . . .
         3  . . . [X] .
         (위치=(3,4))

최종 결과: (r=3, c=4)
```

이 과정에서 핵심은 **"먼저 다음 좌표를 계산 → 격자 안인지 검사 → 통과할 때만 실제로 이동"** 이라는 3단계 패턴이다. 격자를 벗어나는 명령을 거르는 이 검사가 시뮬레이션 문제의 단골 함정이다.

### 방향 벡터(dx, dy) 기법
상하좌우 이동을 `if`문 4개로 쓰면 길고 실수하기 쉽다. 대신 **방향 배열**을 쓰면 깔끔하다.

```
이동 방향을 표로 정리 (행 r, 열 c 기준):
  방향      dr   dc
  R(우)  ->  0  +1
  L(좌)  ->  0  -1
  U(상)  -> -1   0
  D(하)  -> +1   0

  현재 (r,c) 에서
     nr = r + dr
     nc = c + dc
  로 다음 칸을 한 줄에 계산한다.
```

## C++ 예제 코드

### 예제 1: 상하좌우 시뮬레이션 (방향 벡터 사용)

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    string plans;          // 예: "RRRUDD" (공백 없이 들어온다고 가정)
    cin >> n;
    cin >> plans;

    // 시작 위치 (1,1). 행 r, 열 c 모두 1부터 N까지.
    int r = 1, c = 1;

    // 명령 문자와 (dr, dc)를 짝지어 둔다.
    // moveTypes[i] 의 명령을 만나면 dr[i], dc[i] 만큼 이동.
    string moveTypes = "LRUD";
    int dr[4] = {0, 0, -1, 1};   // L, R, U, D 순서
    int dc[4] = {-1, 1, 0, 0};

    for (char ch : plans) {
        // 1단계: 어떤 방향인지 찾는다.
        int dir = -1;
        for (int i = 0; i < 4; i++) {
            if (ch == moveTypes[i]) { dir = i; break; }
        }
        if (dir == -1) continue;  // 알 수 없는 문자는 건너뜀(방어 코드)

        // 2단계: 다음 좌표를 "미리" 계산만 한다 (아직 이동 X)
        int nr = r + dr[dir];
        int nc = c + dc[dir];

        // 3단계: 격자(1..n)를 벗어나면 이 명령은 무시
        if (nr < 1 || nr > n || nc < 1 || nc > n) continue;

        // 통과했을 때만 실제로 이동
        r = nr;
        c = nc;
    }

    cout << r << ' ' << c << '\n';  // 최종 위치 출력
    return 0;
}
```

### 예제 2: 완전 탐색 — "시각" 중 숫자 3이 하나라도 포함된 경우 세기
하루(0시 0분 0초 ~ N시 59분 59초)의 모든 시각을 직접 다 만들어 보고, 어딘가에 `3`이 들어 있으면 카운트한다. 경우의 수가 최대 24×60×60 = 86,400 이라 전부 돌려도 충분히 빠르다.

```cpp
#include <bits/stdc++.h>
using namespace std;

// 주어진 (h, m, s) 안에 숫자 3이 하나라도 있으면 true
bool hasThree(int h, int m, int s) {
    // 각 자리를 문자열로 합쳐 한 번에 검사하면 간단하다.
    string t = to_string(h) + to_string(m) + to_string(s);
    for (char ch : t) {
        if (ch == '3') return true;
    }
    return false;
}

int main() {
    int n;            // 0 <= n <= 23
    cin >> n;

    int count = 0;
    // 모든 시각을 빠짐없이 순회 (완전 탐색)
    for (int h = 0; h <= n; h++) {
        for (int m = 0; m < 60; m++) {
            for (int s = 0; s < 60; s++) {
                if (hasThree(h, m, s)) {
                    count++;
                }
            }
        }
    }

    cout << count << '\n';
    return 0;
}
```

### 예제 3: 격자 회전 (2D 배열 다루기 패턴)
시뮬레이션에서 자주 나오는 "행렬을 시계 방향 90도 회전" 동작이다.

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n = 3;
    // 원본 격자
    vector<vector<int>> a = {
        {1, 2, 3},
        {4, 5, 6},
        {7, 8, 9}
    };

    // 결과 격자 (크기 동일, n x n 일 때)
    vector<vector<int>> b(n, vector<int>(n));

    // 시계 방향 90도 회전 공식:
    //   새 위치 b[j][n-1-i] = 원래 a[i][j]
    // (왜? 윗줄이 오른쪽 열로 가야 하므로)
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            b[j][n - 1 - i] = a[i][j];
        }
    }

    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) cout << b[i][j] << ' ';
        cout << '\n';
    }
    // 출력:
    // 7 4 1
    // 8 5 2
    // 9 6 3
    return 0;
}
```

## 시간복잡도
구현 문제에는 정해진 단일 복잡도가 없다. "내가 적은 절차의 양"이 그대로 복잡도가 된다.

- **상하좌우 시뮬레이션**: 명령 길이 L에 대해 O(L). (각 명령을 한 번씩 처리)
- **완전 탐색(시각 문제)**: 경우의 수만큼 → O(24 × 60 × 60) ≈ O(상수). 일반화하면 모든 후보를 K개 만들면 O(K).
- **격자 처리(회전/순회)**: N×N 격자를 모두 훑으면 O(N²).
- **일반 규칙**: "내가 도는 반복문 횟수의 곱"이 곧 복잡도다.

> 완전 탐색을 시도하기 전 반드시 **경우의 수를 먼저 손으로 계산**하라. 보통 연산이 약 10^7~10^8 이하면 1~2초 안에 통과한다. 그보다 크면 구현이 아니라 다른 알고리즘(그리디/DP)이 필요하다는 신호다.

## 흔한 실수 / 팁

**1. 인덱스가 0부터인지 1부터인지 헷갈림**
문제는 (1,1)부터인데 배열은 0부터라 자꾸 어긋난다. 시작할 때 "이 문제는 1-base다"라고 정해 놓고 경계 검사(`nr < 1 || nr > n`)를 거기에 맞춰라. 혼용하지 말 것.

**2. 경계(격자 밖) 검사를 빼먹음**
시뮬레이션 오답의 1순위. **"이동 → 검사" 가 아니라 "다음 칸 계산 → 검사 → 이동"** 순서를 지켜라. 검사 전에 먼저 좌표를 바꿔 버리면 배열 범위를 벗어나 런타임 에러가 난다.

**3. 부등호 실수 (`<` vs `<=`)**
`for (s = 0; s < 60; s++)` 처럼 "미만/이하"를 항상 의심하라. 60초는 존재하지 않으므로 `< 60`이 맞다. 0~n시는 `<= n`이 맞다.

**4. 방향 배열 dx/dy의 순서를 코드 곳곳에서 다르게 씀**
`dr`, `dc`의 인덱스 순서(예: L,R,U,D)를 한 곳에서 정했으면 코드 전체에서 동일하게 사용하라. 순서가 어긋나면 위로 가야 할 게 아래로 간다.

**5. 문제 조건을 끝까지 안 읽음**
구현 문제는 조건이 길고 예외(특수 규칙, "단, ~인 경우는 제외")가 많다. 한 줄이라도 빠뜨리면 통째로 틀린다. **조건을 한 줄씩 체크리스트로 만들어** 코드와 1:1로 대응시켜라.

**6. 큰 수 / 자료형 오버플로**
누적 합·곱이 int(약 21억) 범위를 넘을 것 같으면 `long long`을 써라. 시간(초)·좌표 누적에서 종종 터진다.

**팁 — 코드를 작게 나누기**: `hasThree()`처럼 "한 가지 판단"을 함수로 빼면 메인 반복문이 읽기 쉬워지고 디버깅이 쉬워진다. 구현이 길어질수록 함수 분리가 강력한 무기다.

**팁 — 시뮬레이션은 손으로 먼저**: 작은 예제를 종이에 직접 한 스텝씩 굴려 보고, 그 절차를 그대로 코드로 옮겨라. "위 동작 과정 ASCII 그림"을 머릿속에 그릴 수 있으면 절반은 푼 것이다.
