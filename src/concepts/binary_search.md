# 이진 탐색 (Binary Search)

## 한 줄 요약
정렬된 데이터에서 매번 탐색 범위를 절반으로 줄여, O(log N)의 속도로 원하는 값(혹은 그 경계)을 찾는 알고리즘이다.

## 언제 쓰나
- **정렬된 배열에서 특정 값이 있는지/몇 번째에 있는지** 찾기 (예: 사전, 전화번호부 검색)
- **"조건을 만족하는 최소/최대 값"을 구하는 최적화 문제** — 이른바 "정답을 이분 탐색" (Parametric Search)
  - 예: "랜선을 N개 이상 만들 수 있는 가장 긴 길이는?" (백준 1654 랜선 자르기)
  - 예: "나무를 잘라 M미터를 얻기 위한 절단기 높이의 최댓값은?" (백준 2805 나무 자르기)
  - 예: "각 집 사이 거리가 최소가 되도록 공유기 C개 설치" (백준 2110 공유기 설치)
- **값이 단조 증가/감소하는 함수에서 특정 결과를 만드는 입력 찾기**
- **lower_bound / upper_bound**: 정렬된 배열에서 "X 이상인 첫 위치", "X 초과인 첫 위치" 구하기
- **회전된 정렬 배열, 봉우리(peak) 찾기** 등 변형 문제

핵심 신호: **데이터가 정렬되어 있거나**, **"답이 커질수록 조건이 한 방향으로만 바뀐다(단조성)"**면 이진 탐색을 의심하라.

## 왜 이걸 쓰나 + 대안과 차이

| 방법 | 시간복잡도 | 전제 조건 | 비고 |
|------|-----------|----------|------|
| **선형 탐색 (for문)** | O(N) | 없음 | 정렬 불필요, 작은 데이터엔 충분 |
| **이진 탐색** | O(log N) | **정렬 필요** | N이 1억이어도 약 27번이면 끝 |
| **해시(unordered_set/map)** | 평균 O(1) | 없음 | "존재 여부"만 빠르게, 단 "X 이상" 같은 경계 질의는 불가 |
| **이진 탐색 트리(set/map)** | O(log N) | 없음(트리가 정렬 유지) | 삽입/삭제가 잦을 때 |

- **선형 탐색과의 차이**: N=1,000,000,000일 때 선형은 최악 10억 번, 이진 탐색은 약 30번. 압도적이다.
- **해시와의 차이**: 해시는 "값이 있냐?"엔 가장 빠르지만, **"X 이상인 값들"이나 "정렬 순서상 위치"** 같은 범위/경계 질의는 못 한다. 이진 탐색은 정렬된 순서를 활용해 이런 경계 질의가 강력하다.
- **이진 탐색의 진짜 강점**: 단순히 배열에서 값을 찾는 것뿐 아니라, **"답의 범위"를 두고 이분 탐색하는 Parametric Search**가 코딩테스트의 단골이다. 답 후보가 [1, 10^9]처럼 넓어도 log 스케일로 빠르게 좁힌다.

왜 정렬이 필수인가? 이진 탐색은 "중간값을 보고 찾는 값이 왼쪽인지 오른쪽인지" 판단해 절반을 버린다. 정렬되어 있어야 "중간보다 작으면 무조건 왼쪽"이라는 보장이 생긴다.

## 동작 과정

### 예제 1: 값 7을 찾기 (기본 이진 탐색)

정렬된 배열에서 `target = 7`을 찾는다. `lo`, `hi`로 범위를 표시하고 `mid = (lo+hi)/2`를 본다.

```
배열(인덱스):  0   1   2   3   4   5   6   7   8
값         : [ 1   3   5   7   9  11  13  15  17 ]

[1단계] lo=0, hi=8, mid=4  ->  arr[4]=9
        lo                  mid                  hi
        v                    v                    v
      [ 1   3   5   7  (9)  11  13  15  17 ]
        9 > 7  이므로 target은 왼쪽에 있다.  hi = mid-1 = 3
        오른쪽 절반( 9 11 13 15 17 ) 버림!

[2단계] lo=0, hi=3, mid=1  ->  arr[1]=3
        lo      mid      hi
        v        v        v
      [ 1  (3)   5    7 ]  9  11  13  15  17
        3 < 7  이므로 target은 오른쪽에 있다.  lo = mid+1 = 2
        왼쪽 절반( 1 3 ) 버림!

[3단계] lo=2, hi=3, mid=2  ->  arr[2]=5
              lo/mid  hi
                v      v
        1   3  (5)     7
        5 < 7  이므로 오른쪽.  lo = mid+1 = 3

[4단계] lo=3, hi=3, mid=3  ->  arr[3]=7
                     lo/mid/hi
                         v
        1  3  5         (7)
        7 == 7  찾았다!  인덱스 3 반환.
```

범위가 9개 -> 4개 -> 2개 -> 1개로 매번 반토막 나는 것이 핵심이다. 9개를 단 4번 만에 끝냈다.

### 예제 2: lower_bound (값 6 이상인 첫 위치 찾기)

값이 정확히 없을 수도 있다. "6 이상인 첫 위치"를 찾는 경계 탐색을 보자.

```
배열      : [ 1   3   5   7   9  11 ]   target = 6 이상인 첫 위치?

답이 들어갈 수 있는 위치는 0..6 (끝 다음 포함). [lo, hi) 반열린 구간 사용.

[1단계] lo=0, hi=6, mid=3  ->  arr[3]=7
        7 >= 6  (조건 만족) -> 더 왼쪽에도 답 있을 수 있다. hi = mid = 3
[2단계] lo=0, hi=3, mid=1  ->  arr[1]=3
        3 >= 6 ? No        -> 오른쪽으로. lo = mid+1 = 2
[3단계] lo=2, hi=3, mid=2  ->  arr[2]=5
        5 >= 6 ? No        -> lo = mid+1 = 3
[4단계] lo=3, hi=3  ->  lo == hi 이므로 종료. 답 = 3

        인덱스 3 (값 7)이 "6 이상인 첫 원소". 정답!
```

## C++ 예제 코드

```cpp
#include <bits/stdc++.h>
using namespace std;

// ---------------------------------------------------------
// 1) 기본 이진 탐색: target의 인덱스를 찾는다. 없으면 -1.
//    [lo, hi] 닫힌 구간 방식.
// ---------------------------------------------------------
int binarySearch(const vector<int>& arr, int target) {
    int lo = 0, hi = (int)arr.size() - 1;
    while (lo <= hi) {                 // 구간에 원소가 남아 있는 동안
        int mid = lo + (hi - lo) / 2;  // (lo+hi)/2 대신 -> 오버플로 방지
        if (arr[mid] == target) {
            return mid;                // 찾음
        } else if (arr[mid] < target) {
            lo = mid + 1;              // 오른쪽 절반만 남김
        } else {
            hi = mid - 1;              // 왼쪽 절반만 남김
        }
    }
    return -1;                         // 못 찾음
}

// ---------------------------------------------------------
// 2) lower_bound 직접 구현: "target 이상"인 첫 인덱스.
//    없으면 arr.size() 반환. [lo, hi) 반열린 구간 방식.
// ---------------------------------------------------------
int myLowerBound(const vector<int>& arr, int target) {
    int lo = 0, hi = (int)arr.size();  // hi는 "끝 다음"까지 포함
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (arr[mid] >= target) {
            hi = mid;        // 조건 만족 -> 답 후보, 더 왼쪽 탐색
        } else {
            lo = mid + 1;    // 조건 불만족 -> 오른쪽 탐색
        }
    }
    return lo;  // lo == hi 일 때가 정답 위치
}

// ---------------------------------------------------------
// 3) Parametric Search 예시: 나무 자르기 (백준 2805 스타일)
//    높이 H로 자를 때 얻는 나무 총량 >= M 인 "최대 H"를 찾는다.
//    H가 커질수록 얻는 양은 단조 감소 -> 이분 탐색 가능.
// ---------------------------------------------------------
long long getWood(const vector<long long>& trees, long long h) {
    long long sum = 0;
    for (long long t : trees)
        if (t > h) sum += (t - h);   // h보다 높은 부분만 잘림
    return sum;
}

long long maxCutHeight(const vector<long long>& trees, long long M) {
    long long lo = 0, hi = *max_element(trees.begin(), trees.end());
    long long ans = 0;
    while (lo <= hi) {
        long long mid = lo + (hi - lo) / 2;
        if (getWood(trees, mid) >= M) {
            ans = mid;       // mid에서 충분히 얻음 -> 더 높여본다
            lo = mid + 1;
        } else {
            hi = mid - 1;    // 부족 -> 더 낮춰야 한다
        }
    }
    return ans;
}

int main() {
    vector<int> arr = {1, 3, 5, 7, 9, 11, 13, 15, 17};

    cout << binarySearch(arr, 7)  << "\n";  // 3
    cout << binarySearch(arr, 8)  << "\n";  // -1
    cout << myLowerBound(arr, 6)  << "\n";  // 3 (값 7의 위치)

    // STL을 쓰면 직접 구현 없이 한 줄로도 가능:
    auto it = lower_bound(arr.begin(), arr.end(), 6);
    cout << (it - arr.begin()) << "\n";     // 3

    vector<long long> trees = {20, 15, 10, 17};
    cout << maxCutHeight(trees, 7) << "\n"; // 15
    return 0;
}
```

> 실무 팁: 코딩테스트에서 단순 "정렬된 배열에서 값/경계 찾기"는 굳이 직접 구현 말고
> `std::lower_bound`, `std::upper_bound`, `std::binary_search`를 쓰면 빠르고 안전하다.
> 직접 구현은 Parametric Search처럼 "조건 함수"가 들어가는 문제에서 필요하다.

## 시간복잡도
- **탐색**: O(log N). N=10억이어도 log2(10억) ≈ 30번이면 끝난다.
- **공간**: O(1) (반복문 구현 기준). 재귀로 짜면 호출 스택 O(log N).
- **단, 정렬이 안 되어 있으면 정렬 비용 O(N log N)이 선행**된다. 한 번 정렬해두고 여러 번 탐색할 때 이득이 크다.
- Parametric Search: O(log(범위) × 판정함수비용). 위 나무 자르기는 O(log(maxHeight) × N).

## 흔한 실수 / 팁
- **정렬 안 된 배열에 이진 탐색**: 가장 흔한 버그. 반드시 정렬 선행. (또는 단조성 확인)
- **`mid = (lo + hi) / 2` 오버플로**: lo, hi가 int 최댓값 근처면 합이 오버플로된다. **`mid = lo + (hi - lo) / 2`**로 쓰자.
- **무한 루프**:
  - `[lo, hi]` 닫힌 구간이면 종료 조건은 `while (lo <= hi)`, 갱신은 `lo=mid+1` / `hi=mid-1` (반드시 mid를 포함하지 않게 ±1).
  - `[lo, hi)` 반열린 구간이면 종료 조건은 `while (lo < hi)`, 갱신은 `lo=mid+1` / `hi=mid`. **두 스타일을 섞으면 무한 루프**가 난다. 하나를 정해 일관되게 쓰자.
- **lower_bound vs upper_bound 혼동**:
  - `lower_bound`: target **이상**인 첫 위치 (`>=`)
  - `upper_bound`: target **초과**인 첫 위치 (`>`)
  - "정렬 배열에서 target의 개수" = `upper_bound(x) - lower_bound(x)`.
- **경계(off-by-one)**: 찾는 값이 없을 때 lower_bound는 `arr.size()`(끝 다음)를 반환할 수 있다. 인덱스 접근 전 범위 체크 필수.
- **Parametric Search 방향 헷갈림**: "조건을 만족하면 답을 저장하고 어느 쪽으로 갈지"를 종이에 단조성 표로 그려라. "최댓값을 구하면 만족 시 `lo=mid+1`(더 키워봄), 최솟값을 구하면 만족 시 `hi=mid-1`(더 줄여봄)".
- **답 변수 따로 저장**: Parametric Search에서는 루프 종료 후의 lo/hi 해석이 헷갈리므로, 조건 만족 시점에 `ans = mid`로 따로 저장해두면 안전하다.
- **실수(double) 이진 탐색**: 실수 답을 구할 땐 `lo <= hi` 대신 정해진 반복 횟수(예: 100번)나 `hi - lo > 1e-9`로 종료해야 한다.
